Modelle 🧠

Weise jeder Rolle ein Modell zu und gib die Slot-Zahl an. Effektiver Kontext = ctx_total ÷ Slots (llama-swap --parallel teilt den Kontext). Beim Speichern werden llm_model, vision_model und das Token-Budget (llm_ctx) automatisch gesetzt. Hinweis: Das tatsächliche --parallel auf dem GPU-Server muss dort separat passen — hier wird der bekannte Wert zur Budgetierung hinterlegt.

Integrationen 🔗

Obsidian-Notizen

„Notiz: …“ landet in der Obsidian-Vault des Nutzers. Zuordnung: JARVIS-Nutzername (klein) → absoluter Vault-Pfad auf dem Host.

Kalender

Pro Nutzer ein Kalender + ein gemeinsamer; iCal-Abo nur im LAN. JARVIS verwaltet Termine per Sprache/Chat (add_event/list_events/…).

Nutzermodelle 📇

Fortgeschriebene Profile je Nutzer (Phase 3) — JARVIS pflegt sie automatisch aus den Gesprächen. Hier ansehen, editieren oder leeren.

Autonomie — Automatisierungen

JARVIS handelt hier selbstständig: zeitgesteuert (einmalig/Intervall/täglich/wöchentlich) oder durch Ereignisse (z.B. erkannter Sprecher). Bei Fälligkeit führt der Agent die Aufgabe im Hintergrund aus und meldet das Ergebnis an die Zielquelle — meldet der Agent „SILENT“, passiert nichts.

Bei autonomen Läufen gelten die Rechte des Besitzers, diese Tools/MCP-Server sind aber zusätzlich gesperrt — egal welche Freigaben sonst bestehen.

Selbst-gebaute Skills 🧰

Wiederverwendbare Werkzeuge, die JARVIS selbst aus Python-Code baut und in der gehärteten Sandbox ausführt. Global für alle Nutzer. Du kannst sie hier deaktivieren, den Code bearbeiten (wird vor dem Speichern getestet), testweise ausführen oder löschen.

Geräte / Satelliten

Alle verbundenen Quellen (Browser, Pi- & ESP-Satelliten). Satelliten senden ein Lebenszeichen (Heartbeat) inkl. Raum, Lautstärke und WLAN-Signal.

Debug / Aufzeichnung

Zeichnet alle Vorgänge auf (STT, Turns, LLM-Aufrufe, Tool-/MCP-Aufrufe mit Argumenten/Ergebnis/Dauer, Retries, Fehler). Hilft bei Analyse — im Alltag besser aus.

Code-Sandbox (Tier 3 — eigener Container)
                    
JARVIS kann eigenen Code (Python/Shell) ausführen — isoliert
                       im Container deploy/sandbox, nie auf dem Host. Tools run_python/run_shell
                       sind über Gruppen-Rechte und die Autonomie-Blacklist steuerbar.

                    
                    
                    

                        
                        
                    

                

                
Netzwerkzugriff (fetch_url / Recherche)
                    
Standardmäßig darf fetch_url nur ÖFFENTLICHE Adressen
                       laden (SSRF-Schutz). Für ein Heimnetz kannst du den Zugriff auf interne LAN-Adressen (z.B.
                       192.168.x.x — Router, NAS, Geräteseiten) erlauben. Loopback bleibt gesperrt.

                    
                

                
Messaging-Kanal (Telegram)
                    
Fester, geräteunabhängiger Kommunikationsweg: JARVIS meldet
                       Automatisierungen/Alarme zuverlässig hierher und du kannst ihm über Telegram schreiben.
                       Bot bei @BotFather anlegen → Token eintragen. Eigene Chat-ID: dem Bot /start
                       schreiben (er antwortet mit der ID) und pro Nutzer unter „Nutzer" hinterlegen.

                    
                    
                    
                    

                        
                        
                        
                    

                    
🔒 Sicherheit: JARVIS schreibt
                       und antwortet ausschließlich an verifizierte (zugeordnete) Chat-IDs. Unbekannte
                       Absender werden nur hier als „ausstehend" gelistet — keine Antwort, kein Agentenlauf.
