## #4 — Persistente Observability  (`debug.py`, `app.py`)
**Problem:** `debug` war ein reiner In-Memory-Ringpuffer (1000 Events), nach Neustart weg, ohne
Aggregation → „warum war er gestern verwirrt?" war nicht beantwortbar.

**Umsetzung:** `debug.log(kind, **data)` speist jetzt aus **denselben** Aufrufen drei Ebenen
(kein Call-Site-Umbau):
1. Ring-Puffer (wie bisher, Admin-UI-Toggle).
2. **Persistente JSONL-Logs** unter `logs/events-YYYYMMDD.jsonl` (append-only, immer an, nur wichtige
   Event-Arten: turn/turn_done/tool/llm_error/retry/context_trim/…).
3. **Aggregierte Metriken** (immer an): Turns, Fehlerquote, Ø/Max Turn-Latenz, Tool-Calls je Name,
   LLM-Fehler/502, Retries, Kontext-Trims.

**Neu:** `GET /api/admin/metrics` → `{metrics, recent}` (Snapshot + letzte persistierte Events).
**Verifiziert:** Live `turns=1, turn_ms_avg=5544`; JSONL-Datei wird geschrieben.

## #2 — Persistenter Verlauf + Tool-Ergebnis-Gedächtnis  (`store.py`, `session_hub.py`, `app.py`, `tools.py`)
**Problem:** Kurzzeitgedächtnis lag nur im RAM → bei jedem Neustart weg. Tool-Ergebnisse landeten nie
im Verlauf → Rückfragen zu zuvor Geholtem (z.B. „was stand in der index.js?") scheiterten.

**Umsetzung:**
- Neue Tabelle `chat_history` in Postgres; `session_hub` schreibt **write-through** und **lazy-loaded**
  beim ersten Zugriff nach Neustart. DB-Ausfall bleibt unkritisch (RAM-Fallback).
- **Tool-Ergebnis-Gedächtnis:** `execute_tool` schneidet substanzielle Ergebnisse knapp mit (max. 3
  jüngste, je 1500 Zeichen); `_prepare_turn` blendet sie als „Letzte Werkzeug-Ergebnisse" in den
  System-Prompt ein → Folge-Turn kann darauf antworten ohne erneuten Tool-Aufruf.

**Verifiziert:** Fakt gemerkt → Dienst neu gestartet → frische Instanz lädt ihn aus der DB und
beantwortet die Folgefrage korrekt.

## #3 — Token-bewusstes Kontext-Budget + Rolling-Summary  (`context_budget.py`, `app.py`, `config.py`)
**Problem:** Bisher nur harte Zeichen-Caps pro Tool-Ergebnis. Lange Gespräche konnten das
Kontextfenster trotzdem sprengen (`exceed_context_size_error`).

**Umsetzung:** `context_budget.fit()` budgetiert den GESAMT-Prompt gegen `llm_ctx`:
- Token grob über Zeichen/Token geschätzt (`chars_per_token`, deutsch ~3.0).
- Reserve für Generierung + Tool-Schemas + Overhead (`ctx_reserve_tokens`).
- Passt der Verlauf nicht, werden die **ältesten** Turns getrimmt; ihr Inhalt wandert **extraktiv**
  (latenzfrei, ohne Extra-LLM-Call) in eine **rollierende Zusammenfassung** je Session, die im
  System-Prompt erscheint. `context_trim`-Event wird geloggt.

**Config:** `llm_ctx=40960`, `ctx_reserve_tokens=8000`, `chars_per_token=3.0`, `summary_max_chars=1500`.
**Verifiziert:** Bei künstlich kleinem Budget 6/8 Turns getrimmt + Zusammenfassung erzeugt.

## #7 — Deferred-Loading für MCP-Werkzeuge  (`mcp_hub.py`, `tools.py`, `app.py`)
**Problem:** Alle MCP-Tools aller Server wurden immer als Schema eingeblendet (z.B. 27 Domoticz-Tools)
→ großer, flacher Tool-Namensraum, mehr Tokens, mehr Tool-Verwechslung (`problems.md` #1/#9).
*(Client-Aktionen und Automationen waren bereits über kompakte Meta-Tools deferred — daher hier nur MCP.)*

**Umsetzung (Muster wie Skills):**
- `mcp_hub.catalog_hint()` blendet nur einen **kompakten Katalog** (Server + Tool-Namen) in den
  System-Prompt ein.
- Neue Meta-Tools `search_mcp_tools(query)` und `load_mcp_tools([…])`; geladene Tools landen in
  `ctx["loaded_mcp"]` und werden im Tool-Loop via `mcp_hub.schemas_for(...)` typsicher nachgereicht.
- Denk-Heuristik (`auto`) prüft MCP-Verfügbarkeit jetzt über `mcp_hub.has_servers()` statt über die
  (nicht mehr eingeblendeten) Schemas.
- **Autonome Läufe** behalten bewusst die vollen MCP-Schemas (vordefinierte Aufgaben → Werkzeuge
  direkt verfügbar, höhere Zuverlässigkeit).

**Verifiziert:** Eval „Schalte das Licht im Wohnzimmer ein" → Modell wählt korrekt `load_mcp_tools`.

# Teil 2 — Folgearbeiten (#1, #10, #11, #12)

## #1 — Log-Retention + Privacy  (`debug.py`, `config.py`)
**Problem:** Die in Teil 1 eingeführten JSONL-Logs wuchsen unbegrenzt und enthielten Nutzertext/
Tool-Argumente im Klartext.
**Umsetzung:** `_cleanup_old_logs` löscht `events-*.jsonl` älter als `log_retention_days` (Default 14,
max. 1×/Tag). `_redact` ersetzt bei `log_redact=true` die Felder message/reply/args/result durch
Längenmarker (`<redacted N chars>`) — Diagnose bleibt, PII raus.
**Config:** `log_retention_days=14`, `log_redact=false`. **Test:** `test_log_redaction`.

## #10 — Verify-by-Tool gegen halluzinierten Erfolg  (`app.py`, `tools.py`)
**Problem:** Modell behauptete „gesendet/erledigt", ohne das Werkzeug aufgerufen zu haben (problems.md #8).
**Umsetzung:**
- `execute_tool` schneidet ALLE Tool-Aufrufe des Turns mit (Name + Erfolg via `_looks_failed`).
- `_finalize_turn` prüft die finale Antwort gegen `_CLAIM_RE` (Vollzugs-Verben): Behauptung vorhanden,
  aber kein geglückter Side-Effect-Tool-Aufruf → Event **`unverified_claim`** (geloggt + Metrik).
- Prompt-Regel verschärft: „Stütze die Erfolgsmeldung auf das ECHTE Tool-Ergebnis; bei Fehler ehrlich sein."
- Bewusst nur **messen/melden**, nicht die Nutzerantwort umschreiben (keine False-Positive-Störung im Voice).
**Tests:** `test_tool_failure_detection`, `test_unverified_claim_flagged`. Sichtbar in `/api/admin/metrics`.

## #11 — Fortschritts-/Heartbeat-Events  (`app.py`, `tools.py`, `static/app.js`)
**Problem:** Lange Tool-Loops/Recherchen fühlten sich wie ein Hänger an (keine Rückmeldung).
**Umsetzung:** SSE-Event `status` zeigt „rufe <tool> auf…" / „denke weiter…".
- Im gedachten Stream-Pass direkt geyieldet.
- Im (nicht-streamenden) adaptiven Fast-Pass über einen `status_cb`, den `execute_tool` aufruft, während
  `_run_loop` als Task nebenläufig arbeitet.
- Frontend zeigt den Status in der Hinweiszeile. **Verifiziert:** `status`-Event erscheint beim Fetch-Turn.

## #12 — Barge-in (Abbruch) + Streaming-STT-Bewertung  (`app.py`, `session_hub.py`, `static/app.js`)
**Problem:** Kein Unterbrechen, während Jarvis spricht/denkt.
**Umsetzung (Barge-in vollständig):**
- Cancel-Registry in `session_hub` (`request_cancel`/`is_cancelled`/`clear_cancel`).
- Stream-Loop prüft den Abbruch zwischen Schritten, nach Sätzen **und** per 0,25-s-Queue-Poll (greift auch
  in der stillen Denkphase). Neues SSE-Event `cancelled`.
- Auslöser: `POST /api/chat/cancel {session_id}` und WS-Nachricht `{type:"cancel"}` (Satelliten).
- Frontend: `AbortController` + `bargeIn()` stoppt Wiedergabe/Queue, bricht den Fetch ab und ruft Cancel;
  ausgelöst beim Mikrofon-Klick, während Jarvis noch spricht/antwortet.
- **Verifiziert:** Abbruch nach ~2,1 s mitten in einer 24-s-Antwort.
- **Grenze (dokumentiert):** Ein einzelner langer, blockierender Tool-Aufruf (z.B. 30-s-`research`) wird
  erst zwischen Schritten abgebrochen, nicht mitten im Aufruf.
**Streaming-STT (bewertet, NICHT umgesetzt):** Echte Teil-Transkripte brauchen einen Streaming-ASR-Endpunkt
auf dem GPU-Server (aktuell Batch-`/api/stt` mit faster-whisper). Das ist ein eigener, größerer Schritt
(GPU-seitig) — die Barge-in-Plumbing ist aber bereits da und greift, sobald ein VAD/Streaming-STT triggert.
**Tests:** `test_cancel_registry`; Live-Barge-in-Test im Verlauf.

# Teil 3 — Agenten-Evolution (Hermes-Adaption, Phase 0–4)

Umsetzung von [`roadmap-agent-evolution.md`](../roadmap-agent-evolution.md) außer dem eigenen
Tool-Calling-Modell (Punkt 1, separat). Alle Phasen verifiziert; Unit-Tests in `tests/run_unit.py`.

## Phase 0 — Modell-Registry + Admin-Modellwahl  (`config.py`, `app.py`, `static/admin.*`)
- Registry `models: [{id, role, ctx_total, slots}]` in der Config; Rollen `agent`/`vision`/`subagent`.
- **Effektiver Kontext = ctx_total ÷ slots** (die llama-swap-`--parallel`-Erkenntnis). Helfer
  `config.model_for(role)` / `config.ctx_for(role)`.
- `config.apply_models()` **spiegelt** beim Speichern `llm_model`/`vision_model`/`llm_ctx` → der Rest des
  Codes braucht keine Änderung.
- Endpunkte `GET/POST /api/admin/model-registry`; neuer Admin-Tab **🧠 Modelle** (Dropdown je Rolle,
  Slot-Eingabe, live berechneter effektiver Kontext).
- **Grenze:** Das tatsächliche `--parallel` auf dem GPU-Server bleibt manuell; die Registry treibt das
  Token-Budget. **Test:** `test_model_registry_helpers`. Beispiel: gemma4-12b/2 Slots → ctx 24576.

## Phase 1 — Cross-Session-Recall  (`knowledge.py`, `store.py`, `app.py`, `tools.py`)
- Wortwechsel werden bei `turn_done` **im Hintergrund** embedded (`kind="conversation"`, nur für
  erkannte Nutzer — Gäste nicht, Privacy).
- Tool **`recall_conversation(query)`** findet relevante frühere Gespräche semantisch; Prompt-Hinweis ergänzt.
- **Verifiziert:** „womit rechnet der server" findet den GPU-Wortwechsel (Score 0.66).

## Phase 2 — Selbst-verbessernde Skills  (`skills.py`, `tools.py`, `app.py`, `static/admin.*`)
- `skills.unhealthy(threshold)` (fail_count ≥ N + last_error); `skills.clear_health()`.
- **`repair_skill(name)`**-Tool + `tools.repair_skill_impl`: hinterlegten Fehler + Code ans LLM →
  korrigierter Code, **syntaxgeprüft** übernommen, Fehlerzähler zurückgesetzt; **Code-Änderung setzt das
  Skill auf Sandbox/nicht-autonom zurück** (Re-Review). Endpunkt `POST /api/admin/skills/repair`.
- Admin-UI: „⚠ instabil"-Badge, letzter Fehler, **🔧 Reparieren**-Knopf.
- **Verifiziert:** kaputtes Skill (NameError) automatisch repariert, fail_count→0, trust→sandbox.

## Phase 3 — Agent-kuratiertes Nutzermodell  (`store.py`, `profile.py`, `app.py`, `static/admin.*`)
- Tabelle `user_profile`; Modul `profile.py` **arbeitet** neue Infos per LLM ins bestehende Profil **ein**
  (merge, gekappt) — alle `profile_update_every` (6) Turns im Hintergrund.
- Einspeisung in den System-Prompt des erkannten Sprechers; Endpunkte `GET /api/admin/user-profiles`,
  `POST /api/admin/user-profile`; Admin-Tab **📇 Profile** (ansehen/editieren/leeren).
- Config: `profile_enabled`, `profile_update_every`. **Verifiziert:** aus einem Gespräch entstand ein
  kohärentes Profil (Name/Rolle/Projekt/Vorlieben/Tonfall).

## Phase 4 — Subagenten  (`subagent.py`, `tools.py`)
- **`spawn_subagent(task)`** delegiert eine mehrstufige Teilaufgabe an einen isolierten Mini-Tool-Loop;
  nur das **Endergebnis** kommt zurück (zero-context-cost, schützt das Budget #3).
- Modellwahl aus der Registry (Rolle `subagent`); **Rekursionssperre** (`_MAX_DEPTH=1`) +
  **Nebenläufigkeits-Semaphore** (2); Rechte/Namespace vom Eltern-Lauf geerbt.
- **Verifiziert:** Subagent löste „17×23" → 391; verschachteltes Spawnen wird blockiert.
  **Test:** `test_subagent_recursion_guard`.
